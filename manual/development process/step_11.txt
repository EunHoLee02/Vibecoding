이번 단계 인프라 / 배포 초안 요약:
1. 프론트엔드는 Next.js 특성에 맞춰 Vercel 배포를 1순위 권장안으로 둔다.
2. 백엔드는 Dockerfile 기반 FastAPI 서비스로 패키징하고, Render 같은 Docker 지원 PaaS 또는 단일 VM 배포가 가능하도록 정리한다.
3. DB는 managed PostgreSQL, Redis는 managed Redis를 권장하고, local / dev는 `docker-compose.yml`로 재현 가능하게 둔다.
4. 파일 스토리지는 dev에서는 로컬 볼륨, staging / prod에서는 S3-compatible object storage 권장으로 정리한다.
5. 헬스체크는 프론트 `/api/health`, 백엔드 `/health/live`, `/health/ready` 기준으로 맞춘다.
6. CI 초안은 GitHub Actions 기준으로 프론트 빌드, 백엔드 의존성 설치, migration, health smoke test까지 포함한다.
7. OCR 엔진 운영 기준은 `paddleocr`로 고정한다.

추천 배포 플랫폼과 이유:
1. 프론트엔드
   - 추천: Vercel
   - 이유:
     - 현재 앱이 Next.js App Router 기반이라 배포 호환성이 높다.
     - preview deployment 흐름이 강해서 `feature/PR -> preview`, `main -> production` 분리 운영이 쉽다.
     - 프론트와 백엔드 배포 수명주기를 분리하기 좋다.
2. 백엔드
   - 추천: Render Docker web service
   - 이유:
     - 현재 백엔드는 FastAPI + Alembic + Redis + Postgres 조합이라 Docker 배포와 잘 맞는다.
     - Dockerfile 기준 배포면 local / staging / prod 재현성이 높다.
     - 환경변수 주입, 헬스체크 연결, 롤백 운영이 비교적 단순하다.
3. DB
   - 추천: managed PostgreSQL
   - 우선 권장 예시: Neon
   - 이유:
     - Postgres 호환성이 높다.
     - staging / preview DB 분리에 유리하다.
4. Redis
   - 추천: managed Redis
   - 우선 권장 예시: Upstash
   - 이유:
     - 세션 캐시 용도에 맞고 운영 부담이 낮다.
5. 파일 스토리지
   - 추천:
     - dev / single-node staging: local volume
     - production: Cloudflare R2 또는 다른 S3-compatible object storage
   - 이유:
     - 현재 업로드 흐름은 object storage 전환과 궁합이 좋다.
     - 멀티 인스턴스 운영에서는 로컬 디스크 공유가 안 되므로 외부 스토리지가 필요하다.

프론트엔드 배포 방식:
1. 배포 단위: `apps/frontend`
2. 빌드 명령: `npm run build`
3. 실행 명령: `npm run start -- --hostname 0.0.0.0 --port 3000`
4. 환경 전략:
   - preview: PR/브랜치별 preview URL
   - staging: staging API URL로 별도 빌드
   - production: production API URL로 별도 빌드
5. 운영 주의:
   - `NEXT_PUBLIC_API_BASE_URL`은 클라이언트 번들에 포함되는 build-time 값이다.
   - preview/staging/prod는 반드시 각 환경별 값으로 다시 빌드해야 한다.
   - preview 빌드 산출물을 production에 재사용하면 잘못된 API 주소가 번들될 수 있다.
6. SSR/서버 fetch용 값:
   - `API_BASE_URL`은 서버사이드 fetch용이다.
   - 같은 호스트 구조면 public URL을 써도 되고, 플랫폼이 private service URL을 주면 내부 URL 사용도 가능하다.

백엔드 배포 방식:
1. 배포 단위: `apps/backend`
2. 배포 형식: Docker image
3. 현재 기본 시작 명령:
   - `alembic upgrade head && uvicorn app.main:app --host 0.0.0.0 --port 8000`
4. 운영 권장:
   - 초기 P0 운영에서는 위 방식 사용 가능
   - 운영 규모가 커지면 migration step과 app startup step을 분리
5. 운영 주의:
   - OCR 파일이 로컬 스토리지에 남는 구조이므로 multi-instance autoscale 전에는 object storage 전환이 필요하다.

DB 호스팅 방식:
1. dev:
   - `docker-compose`의 Postgres 사용
2. staging:
   - managed PostgreSQL 별도 인스턴스
   - production과 분리된 DB
3. prod:
   - managed PostgreSQL
   - 자동 백업 / PITR 가능한 플랜 권장
4. migration 전략:
   - 현재 초안은 backend boot 전에 `alembic upgrade head` 수행
   - production에서 인스턴스 수가 늘어나면 release job 또는 one-off migration command로 분리
   - destructive migration 전에는 snapshot / restore point 확보

파일 스토리지 방식:
1. dev:
   - backend container volume mount
2. staging:
   - single-node면 volume mount 허용
   - multi-instance 검토 시점부터 R2 전환 우선
3. prod:
   - S3-compatible object storage 권장
   - presigned upload, source delete, OCR fetch 경로를 object storage 기반으로 이관
4. 추가 운영 메모:
   - 현재 `R2Client`는 로컬 shim이라 이름과 실제 동작이 다르다.
   - production 멀티 인스턴스에 그대로 쓰면 업로드/OCR 파일 정합성 문제가 생길 수 있다.

추가 확인 필요 항목:

충돌 가능 항목:
1. 8단계/9단계 이력 문서를 runtime 기준으로 그대로 해석하면 OCR 엔진 혼선이 생길 수 있다.

3. frontend `NEXT_PUBLIC_API_BASE_URL`은 build-time 값이라 환경별 빌드 분리를 잘못하면 잘못된 API 주소가 번들될 수 있다.
4. migration을 app boot와 같이 실행하는 방식은 초기 배포에는 편하지만, production 운영 규모가 커지면 별도 migration step 분리가 필요할 수 있다.
