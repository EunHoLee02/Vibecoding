아래는 **4단계 IA 및 네비게이션 설계 최종 확정본**입니다.
이번 응답은 **4단계만** 다룹니다.

3단계에서 확정된 주요 화면 구조, 등록→분석 전 확인→분석 진행/재사용→결과 상세 흐름, 회원가입 후 자동 로그인 없음, OCR 중간 처리 독립 화면, 재시도 UX, 관리자 UI 없음 기준을 반영했습니다. 또한 이번 대화에서 추가로 확정한 `supplements 목록 전용 화면`, `모바일 햄버거 메뉴`, `로그인 후 직전 페이지 복귀`, `문의 내역 목록 화면`, `문의 상세 화면 미구현`을 함께 반영했습니다.

# 충돌 항목

1. **이전 4단계 초안의 로그인 후 `/app` 고정 이동과 현재 확정된 직전 페이지 복귀 정책이 충돌**

   * 현재 확정본에서는 `직전 보호 페이지가 있으면 해당 페이지로 복귀`, `없으면 /app 이동`으로 반영함.

2. **이전 4단계 초안의 문의 접수 중심 구조와 현재 확정된 문의 목록 화면 추가가 충돌**

   * 현재 확정본에서는 `/support/inquiries` 목록 화면을 P0로 포함함.

3. **관리자용 메뉴 분리 요구와 3단계의 관리자 UI 없음이 동시에 존재**

   * 현재 확정본에서는 **사용자용 네비게이션과 운영용 구조를 분리 표기**하되, **관리자 웹 메뉴는 생성하지 않음**으로 정리함.

# 1. 전체 사이트맵

## 1-1. 전체 IA 구조

```text
[공개 영역 / 비로그인]
- /                                비로그인 홈 / 서비스 소개 (P0)
- /signup                          회원가입 (P0)
- /login                           로그인 (P0)

[인증 사용자 영역]
- /app                             내 영양제 홈 / 대시보드 (P0)

- /supplements                     내 영양제 목록 (P0)
  - /supplements/new/image         이미지 업로드 등록 (P0)
  - /supplements/ocr-jobs/:jobId/status   OCR 처리 중 상태 (P0)
  - /supplements/:id/review        OCR 검수 / 등록 확정 (P0)
  - /supplements/new/manual        직접 입력 등록 (P0)
  - /supplements/:id               영양제 상세 (P0)
  - /supplements/:id/edit          영양제 수정 (P0)

- /analyses/history                분석 기록 목록 (P0)
  - /analyses/new/preview          분석 전 확인 (P0)
  - /analyses/runs/:runId/status   분석 진행 / 재사용 결과 분기 (P0)
  - /analyses/:analysisId          분석 결과 상세 (P0)

- /support/inquiries               문의 내역 목록 (P0)
  - /support/inquiries/new         문의 / 오류 신고 접수 (P0)

[운영 영역 / 관리자]
- 별도 관리자 웹 콘솔 없음 (P0)
- 운영 기능은 내부 운영 절차 / 내부 API / 로그 기반 대응
```

## 1-2. IA 설계 원칙

* 로그인 후 메인 허브는 **`/app` 내 영양제 홈**으로 둔다.
* 영양제 영역은 **목록 / 등록 / OCR / 상세 / 수정**으로 분리한다.
* 분석 영역은 **기록 / 사전 확인 / 진행 상태 / 결과 상세**로 분리한다.
* 문의 영역은 **문의 목록 / 문의 접수**까지만 P0에 포함한다.
* 문의 상세 화면(`/support/inquiries/:id`)은 **이번 단계 범위에서 제외**한다.
* 모바일은 **하단 탭바 없이 햄버거 메뉴**를 사용한다.
* 정식 관리자 UI는 두지 않는다. 이는 3단계 확정사항을 그대로 유지한 것이다.

* 등록 영양제 0개:

  * 진입 시 빈 상태와 등록 유도 CTA 노출

  * 분석 기록 메뉴는 노출

  * 다른 메뉴 이동은 허용
  * 관련 영양제 수정 저장은 제한 또는 경고 처리

* 분석 실패:

  * 상태 화면에서 `재시도`와 `문의 접수` CTA 우선 노출

이는 3단계의 빈 상태, 오류 상태, 재시도 UX 기준과 일치한다.

# 3. 관리자용 네비게이션

## 3-1. 관리자용 메뉴 구조

**MVP P0 기준: 별도 관리자용 웹 네비게이션 없음**

* 관리자 상단 메뉴 없음
* 관리자 사이드 메뉴 없음
* `/admin` 계열 라우트 없음
* 사용자 서비스 내 관리자 메뉴 노출 금지

## 3-2. 운영 구조 분리 정책

* 운영 기능은 사용자 IA에 섞지 않는다.
* 문의 운영 처리, 오류 대응, 데이터 수정은 내부 운영 절차 / 내부 API / 로그 기반으로 처리한다.
* 추후 관리자 콘솔이 필요해질 경우 사용자 서비스와 분리된 별도 경로 또는 별도 도메인으로 구성하는 것이 안전하다.

**가정**

* 향후 운영 콘솔이 추가되면 `ops.<domain>` 또는 `/internal/*` 구조가 적합하다.
* 다만 이는 이번 단계 확정 범위에 포함하지 않는다.

이 내용은 3단계의 “정식 관리자 UI는 없다”는 기준을 유지한 것이다.

## 4-2. 로그인 후 이동 정책

### 확정 규칙

1. 사용자가 비로그인 상태에서 보호 페이지 접근 시도

   * 현재 접근 URL을 `returnTo`로 저장

2. 로그인 성공 시

   * `returnTo`가 존재하고 내부 허용 경로이면 해당 페이지로 이동
   * `returnTo`가 없으면 `/app`으로 이동

3. 로그인/회원가입 페이지에서 직접 로그인한 경우

   * 직전 보호 페이지 정보 없으면 `/app` 이동

4. `returnTo`가 외부 URL 또는 비허용 경로면

   * 보안상 무시하고 `/app` 이동

### 적용 예시

* `/supplements/123` 접근 시 로그인 필요 → 로그인 성공 후 `/supplements/123`
* `/analyses/456` 접근 시 로그인 필요 → 로그인 성공 후 `/analyses/456`
* 메인에서 로그인 버튼 클릭 → 로그인 성공 후 `/app`

3단계에서는 로그인 성공 후 S-004로 가는 흐름이 있었으므로, 이번 확정본은 이를 **기본 fallback 경로**로 유지하면서 보호 페이지 복귀를 추가한 구조다.

## 이번 단계 확정 사항

1. IA는 **공개 영역 / 인증 사용자 영역 / 운영 영역**으로 분리한다.
2. 로그인 후 허브 화면은 `/app`으로 둔다.
3. `supplements`는 목록 전용 화면(`/supplements`)을 가진다.
4. 문의 기능은 `/support/inquiries` 목록과 `/support/inquiries/new` 접수 화면까지만 P0에 포함한다.
5. 문의 상세 화면(`/support/inquiries/:id`)은 이번 단계 범위에서 제외한다.
6. 모바일 네비게이션은 하단 탭바 없이 햄버거 메뉴로 확정한다.
7. 로그인 후 이동은 **직전 보호 페이지 복귀 우선, 없으면 `/app` fallback**으로 확정한다.
8. 관리자 웹 메뉴는 두지 않고 운영은 내부 절차/API/로그로 대응한다.

## 다음 단계 입력으로 넘길 핵심 요약

1. **주요 레이아웃 기준**

   * 비로그인: 상단 최소 네비
   * 로그인 데스크톱: 상단 네비 + 사이드 네비
   * 모바일: 햄버거 메뉴
   * 하단 탭바 없음

2. **확정된 핵심 목록 화면**

   * `/supplements`
   * `/analyses/history`
   * `/support/inquiries`

3. **확정된 핵심 프로세스 화면**

   * `/supplements/new/image`
   * `/supplements/ocr-jobs/:jobId/status`
   * `/supplements/:id/review`
   * `/supplements/new/manual`
   * `/supplements/:id`
   * `/supplements/:id/edit`
   * `/analyses/new/preview`
   * `/analyses/runs/:runId/status`
   * `/analyses/:analysisId`
   * `/support/inquiries/new`

4. **로그인 복귀 규칙**

   * 보호 페이지 접근 후 로그인 시 원래 페이지 복귀
   * 복귀 정보 없거나 비정상이면 `/app`

5. **5단계 와이어프레임 우선 대상**

   * `/app`
   * `/supplements`
   * `/supplements/:id`
   * `/supplements/:id/review`
   * `/analyses/history`
   * `/analyses/:analysisId`
   * `/support/inquiries`
   * `/support/inquiries/new`

## 충돌 가능 항목

1. 5단계에서 실제 레이아웃을 단일 컬럼 중심으로 단순화하면 데스크톱 사이드 네비 폭이나 메뉴 배치가 일부 조정될 수 있음
2. 6단계 ERD/API에서 OCR job, supplement, analysis run 엔티티 명칭이 바뀌면 일부 URL 세그먼트가 변경될 수 있음
3. 6단계에서 문의 상태값 설계가 바뀌면 `/support/inquiries` 목록 컬럼과 필터 구조가 일부 조정될 수 있음
4. 향후 운영 답변을 사용자에게 보여줘야 한다는 요구가 생기면 문의 상세 화면이 추가될 수 있음
