# 1. 5단계 반영 기준 요약

## 1-1. 이번 단계에서 확정 반영된 사항

* `/app`은 **카드형 대시보드 구조**로 설계한다.
* 직접 입력/수정 폼의 주요 필드는 **제품명 / 제조사 / 복용 정보 / 메모**로 확정한다.
* 문의 접수에는 **파일 첨부를 포함하지 않는다**.
* 비밀번호 찾기 기능은 **P0 포함**으로 하고, **이메일 링크 방식**을 사용한다.
* 비밀번호 재설정 링크 유효시간은 **5분**으로 한다.
* 영양제 삭제 기능은 **P0 포함**으로 한다.
* 영양제 삭제는 **soft delete** 기준으로 설계한다.
* 삭제된 영양제 상세는 **읽기 전용**으로 제공한다.
* 삭제 복구는 **사용자와 운영자 모두 가능**하게 한다.
* 문의 목록의 상태 필터는 **P1로 미룬다**.

## 1-3. 공통 UI 규칙

* 데스크톱은 **상단 네비 + 좌측 사이드바 + 본문** 구조를 기본으로 한다.
* 모바일은 **햄버거 드로어 + 1열 레이아웃**을 기본으로 한다.
* 하단 탭바는 사용하지 않는다.
* Breadcrumb는 자리만 확보하고 실제 우선순위는 P1로 둔다.
* 목록/상세/상태 화면은 모두 **빈 상태 / 오류 상태 / 로딩 상태**를 포함한다.
* 제출형 화면은 가능한 경우 **sticky action bar**를 둔다.
* 상태 처리 화면은 **상태 메시지 + 진행 표시 + 보조 안내 + 재시도/문의 CTA** 패턴을 사용한다.

## W-01. 로그인 `/login` [P0]

### 레이아웃 구조

```text id="owpbok"
[Top Header]
  - Logo
  - Sign up

[Centered Auth Card]
  - Title: 로그인
  - Description
  - Email Input
  - Password Input
  - Error Summary
  - Primary Button: 로그인
  - Text Link: 비밀번호 찾기
  - Secondary Link: 회원가입

[Toast Layer]
```

### 헤더

* 로고 → `/`
* 회원가입 링크

### 사이드바

* 없음

### 본문 영역

* 중앙 정렬 로그인 카드
* `returnTo`가 있는 경우 로그인 후 복귀 안내 문구 표시

### 주요 카드/테이블/리스트

* 로그인 카드 1개

### 입력 폼

* 이메일
* 비밀번호

### 버튼

* 로그인
* 비밀번호 찾기
* 회원가입으로 이동

### 모달/토스트

* 로그인 실패 토스트
* 세션 만료 안내 토스트

### 빈 상태 / 오류 상태 / 로딩 상태

* 빈 상태: 해당 없음
* 오류 상태: 인증 실패 / 네트워크 오류 / 서버 오류
* 로딩 상태: 로그인 버튼 로딩, 폼 비활성화

### 주요 인터랙션

* Enter 제출
* 로그인 성공 시 `returnTo` 우선, 없으면 `/app`
* 로그인된 사용자가 진입 시 `/app` 또는 유효한 내부 경로로 이동

# 7. 5단계 설계 메모

## 7-1. `/app` 카드형 대시보드 의미

`/app`은 단순 목록 페이지가 아니라, 로그인 직후 사용자가 현재 상태를 빠르게 파악하고 다음 행동을 바로 선택할 수 있게 하는 **허브형 홈 화면**이다. 따라서 카드 구조로 설계하여 `빠른 실행`, `상태 요약`, `최근 활동`, `처리 필요 항목`을 분리한다.

## 7-2. 삭제 정책 반영 방식

* 삭제 버튼은 P0로 포함한다.
* 삭제는 soft delete 기준으로 설계한다.
* 삭제된 영양제는 일반 목록에서 숨긴다.
* 분석 결과에서는 이름 유지 + `삭제됨` 배지로 문맥을 보존한다.
* 삭제된 영양제 상세는 읽기 전용이며, 사용자와 운영자 모두 복구 가능하다.

## 7-3. 비밀번호 재설정 정책 반영 방식

* 비밀번호 찾기는 이메일 링크 방식이다.
* 링크는 5분만 유효하다.
* 만료 시 재요청 화면으로 유도한다.

# 이번 단계 확정 사항

1. 5단계 텍스트 와이어프레임 화면 범위를 최종 확정했다.
2. `/app`은 카드형 대시보드 구조로 확정했다.
3. 직접 입력/수정/OCR 검수의 주요 필드를 `제품명 / 제조사 / 복용 정보 / 메모`로 확정했다.
4. 비밀번호 찾기와 비밀번호 재설정 화면을 P0에 포함했다.
5. 비밀번호 재설정은 이메일 링크 방식, 링크 유효시간은 5분으로 확정했다.
6. 영양제 삭제 기능을 P0에 포함했고 soft delete 기준으로 확정했다.
7. 삭제된 영양제 상세는 읽기 전용으로 확정했다.
8. 삭제 복구 권한은 사용자와 운영자 모두 가능하도록 확정했다.
9. 문의 접수는 첨부파일 없이 텍스트 기반으로 확정했다.
10. 문의 목록 상태 필터는 P1로 미뤘다.

# 다음 단계 입력으로 넘길 핵심 요약

1. 6단계 ERD 후보 엔티티

   * user
   * supplement
   * supplement_ocr_job
   * analysis
   * analysis_run
   * inquiry
   * password_reset_token 또는 동등한 인증 토큰 엔티티
2. supplement는 `deleted_at` 기반 soft delete가 필요하다.
3. 삭제된 supplement는 일반 목록 제외, 읽기 전용 상세, 복구 가능 정책이 필요하다.
4. analysis 결과에는 삭제된 supplement 표시를 위한 이름/상태 보존 전략이 필요하다.
5. 비밀번호 재설정은 이메일 링크 방식이며 토큰 만료시간은 5분이다.
6. inquiry는 첨부파일 없이 텍스트 기반 접수만 지원한다.
7. inquiry 목록의 상태 필터는 P1로 제외 가능하다.
8. P0 API 범위에는 아래가 포함되어야 한다.

   * 로그인 / 회원가입
   * 비밀번호 찾기 / 재설정
   * 영양제 목록 / 상세 / 생성 / 수정 / 삭제 / 복구
   * OCR 등록 / OCR 상태 / OCR 검수 저장
   * 분석 전 확인 / 실행 / 실행 상태 / 결과 / 기록
   * 문의 목록 / 문의 접수
