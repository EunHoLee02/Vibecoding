# 1단계 PRD 최종 확정본

## 1. 서비스 개요

### 1-1. 서비스 기본 정보

* **서비스명**: Well Track
* **서비스 형태**: 웹 서비스
* **한 줄 소개**: 사용자가 복용 중인 영양제의 성분과 함량을 사진 또는 직접 입력하면, 중복/과다 복용 가능성, 부족 가능 성분, 복용 시간, 조합 가이드를 제공하는 영양제 복용 관리 웹서비스
* **주요 사용자**: 건강을 챙기고 싶은 일반 사용자
* **사용 환경**: 웹
* **인증 방식**: 이메일 + 비밀번호
* **파일 업로드**: 이미지 업로드 필요
* **외부 연동**: OCR/텍스트 추출용 AI API, 설명 생성용 생성형 AI API
* **수익모델**: 무료
* **결제 기능**: 없음
* **관리자 페이지**: 정식 관리자 페이지 없음
* **개인정보 수집 범위**: 이메일, 이름
* **분석 결과 저장 방식**: 사용자 계정 기준 누적 저장

### 1-2. 서비스 정의

Well Track는 사용자가 복용 중인 여러 영양제를 제품 단위가 아닌 **성분 단위**로 통합 관리할 수 있게 돕는 서비스다. 사용자는 이미지 업로드 또는 직접 입력으로 영양제 정보를 등록하고, 서비스는 국내 기준 영양성분 기준과 내부 분석 룰을 바탕으로 다음 정보를 제공한다. 

* 중복 성분 분석
* 과다 복용 가능성 분석
* 부족 가능 성분 안내
* 복용 시간 추천
* 같이 먹으면 좋은 조합 가이드
* 같이 먹을 때 주의할 조합 가이드
* 목적 기반 추천

  * 예: 피로, 운동, 수면, 눈 건강, 면역/감기 대비

---

## 2. 문제 정의

### 2-1. 사용자가 겪는 핵심 문제

1. 여러 영양제를 함께 복용할 때 **성분 중복 여부를 직접 계산하기 어렵다**.
2. 제품별 표기 방식이 달라 **총 복용량을 한눈에 파악하기 어렵다**.
3. 1회분/1일분 기준이 제각각이라 **과다 복용 가능성을 판단하기 어렵다**.
4. 복용 시간, 식전/식후, 좋은 조합/주의 조합을 **기억하고 관리하기 어렵다**.
5. 식단/생활습관 정보를 입력하지 않은 상태에서도, 현재 복용 조합만 기준으로 **부족 가능 성분의 방향성 정도는 알고 싶다**.

### 2-2. 기존 대안의 한계

* 제품 라벨을 수기로 비교해야 해서 번거롭다.
* 검색 기반 정보는 제품별로 흩어져 있어 성분 총합 관리가 어렵다.
* 일반 건강정보는 많지만, **현재 내가 먹는 조합 기준**으로 바로 해석해주지 못한다.

---

## 3. 목표

### 3-1. 제품 목표

1. 사용자가 복용 중인 영양제를 한 번에 등록하고 **성분 기준으로 통합 분석**할 수 있게 한다.
2. 사용자가 **중복 성분과 과다 복용 가능성**을 빠르게 인지하도록 한다.
3. 사용자가 **복용 시간과 기본 조합 가이드**를 쉽게 이해하도록 돕는다.
4. 분석 결과를 누적 저장하여 **지속적인 복용 관리**가 가능하도록 한다.
5. 결과 설명은 **생성형 AI 답변 수준의 쉬운 설명**으로 제공하되, 판정 자체는 기준 데이터와 내부 룰 기반으로 유지한다.

### 3-2. 사용자 목표

* 내가 먹는 영양제를 한 번에 정리하고 싶다.
* 어떤 성분이 겹치는지 알고 싶다.
* 줄여야 할지 유지해야 할지 빠르게 판단하고 싶다.
* 언제 먹는 것이 좋은지 가이드가 필요하다.
* 부족 가능 성분도 참고 수준으로 확인하고 싶다.

### 3-3. 비즈니스/운영 목표

* 무료 서비스 기준으로 운영 가능한 MVP를 만든다.
* 핵심 판정은 룰 기반으로 유지해 AI 비용 의존도를 낮춘다.
* 향후 AI 설명 고도화, 개인화, 건강설문, 추천 확장이 가능한 구조를 만든다.

## 4. 사용자 페르소나

### 페르소나 A: 20대 건강/운동 관심형

* 운동, 활력, 체형 관리 목적으로 영양제를 3~5개 이상 복용
* 제품별 성분이 겹쳐도 총량을 계산하지 못함
* 운동, 피로, 면역/감기 대비 같은 목적 중심으로 관리하고 싶어함

### 페르소나 B: 50대 건강관리형

* 건강검진 이후 여러 영양제를 복수 복용
* 어떤 성분이 겹치는지, 언제 먹어야 하는지 헷갈림
* 같이 먹어도 되는지, 줄여야 할 제품이 무엇인지 알고 싶어함

---

## 5. 핵심 가치 제안

1. **제품 중심이 아니라 성분 중심으로 관리**

   * 복수 제품을 성분 단위로 통합 분석

2. **중복/과다 가능성을 빠르게 파악**

   * 어떤 성분이 몇 개 제품에서 겹치는지, 총량이 어느 수준인지 표시

3. **복용 시간과 조합 가이드를 한 번에 제공**

   * 식전/식후, 아침/저녁, 기본 조합 주의사항까지 함께 안내

4. **입력 편의성과 정확도 보완을 동시에 확보**

   * 이미지 업로드로 시작하되, 사용자 확인/수정 절차로 오인식 보완

5. **저장형 서비스**

   * 결과를 누적 저장하고 재조회할 수 있음

6. **이해하기 쉬운 설명**

   * 생성형 AI 답변 수준의 자연스러운 설명 제공
   * 단, 의료 진단/처방처럼 해석될 수 있는 표현은 금지

---

## 6. MVP 범위

### 6-1. MVP 원칙

* 핵심은 **복용 관리와 분석**
* 의료 진단/처방 서비스처럼 보이지 않도록 제한
* 판정 로직은 **국내 기준 영양성분 기준 + 내부 룰 기반**
* AI는 **입력 보조 및 설명 보조** 역할로 제한

### 6-2. 사용자 기능 MVP

1. 이메일 회원가입 / 로그인 / 로그아웃
2. 영양제 등록

   * 이미지 업로드
   * 직접 입력
3. 등록 정보 수정

   * 제품명
   * 성분명
   * 함량
   * 1회분 섭취량
   * 1일 섭취량
4. OCR/AI 추출 결과 확인 및 수정
5. 성분 표준화 및 총량 계산
6. 중복 성분 분석
7. 과다 복용 가능성 분석
8. 복용 시간 추천
9. 좋은 조합 / 주의 조합 기본 가이드
10. 목적 기반 추천

    * 피로, 운동, 수면, 눈 건강, 면역/감기 대비
11. 분석 결과 저장 및 과거 결과 조회
12. 결과 요약 가이드 제공

    * 줄이기
    * 유지
    * 검토 필요

### 6-3. 관리자 기능 MVP

* **정식 관리자 페이지는 포함하지 않음**
* 운영자는 별도 내부 방식으로 아래를 처리

  * 문의/오류 접수 확인
  * 기준 데이터 갱신
  * 분석 룰 점검
  * 장애 대응

## 9. 비기능 요구사항

1. **반응형 웹 지원**

   * 모바일 웹, 태블릿, 데스크톱 사용 가능

2. **입력 정확도 보완**

   * 이미지 추출 결과는 사용자 확인 후 저장해야 함

3. **분석 응답 성능**

   * 일반 분석 결과는 사용자가 체감상 불편하지 않은 시간 내 제공
   * **가정**: 핵심 분석 결과는 수 초 내 제공 목표

4. **가용성**

   * 외부 AI API 장애 시에도 직접 입력 기반 핵심 분석은 계속 가능해야 함

5. **확장성**

   * 추후 식단/생활습관, 개인화, AI 설명 강화 기능을 추가할 수 있어야 함

6. **감사 가능성**

   * 어떤 입력값으로 어떤 분석 결과가 생성됐는지 추적 가능해야 함

7. **설명 품질 기준**

   * 설명은 **생성형 AI 답변 수준의 쉬운 자연어 설명**으로 제공
   * 판정은 사용자 입력 + 국내 기준 + 내부 룰에 의해 결정
   * AI는 판정 결과를 이해하기 쉽게 풀어쓰는 보조 역할만 수행

8. **근거출처 범위**

   * **A. 사용자 입력 기반 근거**

     * 제품명, 성분명, 함량, 1회분 섭취량, 1일 섭취량, 이미지 추출 결과
   * **B. 내부 기준 데이터 기반 근거**

     * 국내 기준 영양성분 기준
     * 내부 성분 표준화 매핑
     * 복용 시간 룰
     * 조합 주의/권장 룰
   * **C. 일반 건강 가이드 기반 설명**

     * 일반적인 복용 가이드, 식전/식후 설명, 기본 조합 가이드
   * **D. 생성형 AI 요약 설명**

     * A, B, C를 바탕으로 이해하기 쉬운 자연어 설명 생성
   * 사용자 화면에는 아래 수준으로 표시

     * 입력한 영양제 정보 기준
     * 국내 기준 영양성분 기준 기반
     * 일반 건강 가이드 수준의 설명

## 10. 운영/관리 요구사항

### 10-1. 사용자 기능과 운영 기능 구분

#### 사용자 기능

* 회원가입/로그인
* 영양제 등록/수정
* 분석 실행
* 결과 저장/조회

#### 운영 기능

* 서비스 공지 반영
* 문의 응대
* 기준 영양성분 데이터 업데이트
* 분석 룰 수정
* 장애 발생 시 대응

### 10-2. 정식 관리자 기능 필요 여부

* 현재 MVP 기준으로 **정식 관리자 페이지는 불필요**
* 단, 아래 운영 요소는 필요

  1. 문의 접수 수단
  2. 기준 데이터 관리 방식
  3. 분석 룰 변경 절차
  4. 서비스 장애 대응 절차

### 10-3. 예외 상황 운영 원칙

1. OCR 실패 시 직접 입력으로 유도
2. 기준 데이터가 없는 성분은 “분석 제한”으로 표시
3. 외부 AI API 실패 시 핵심 룰 기반 분석은 계속 제공
4. 의료 오인 우려 문구는 운영 정책으로 제한

## 11. 보안/개인정보 요구사항 (정책 수준)

1. **개인정보 최소 수집**

   * 이메일, 이름만 수집
   * 건강 상태, 질환, 약물 정보는 MVP 필수 수집 대상 아님

2. **명확한 고지**

   * 서비스는 일반 건강관리/복용 가이드용 정보 제공 서비스이며, 의료 진단/처방 대체가 아님을 명시

3. **결과 해석 책임 범위**

   * 분석 결과는 참고용 가이드로 제공
   * 이상 증상 또는 복용 불안이 있을 경우 전문가 상담을 권장

4. **업로드 이미지 처리 정책**

   * 업로드 이미지는 분석 목적 범위 내에서만 사용
   * **추가 확인 필요**: 보관 기간 및 삭제 정책

5. **계정 보안 원칙**

   * 비밀번호는 안전하게 저장해야 함
   * 인증 정보는 최소 권한 원칙으로 관리

6. **로그 정책**

   * 민감한 건강 추론 정보는 과도하게 로그에 남기지 않음

7. **표현 정책**

   * “감기 치료”, “질환 개선 보장”, “의학적 처방” 등으로 오해될 수 있는 표현 금지
   * “면역/감기 대비 목적의 일반 가이드” 수준으로 제한

---

## 12. KPI

### 12-1. 확정 KPI

1. **첫 달 분석 완료자 수**

   * 정의: 회원가입 후 실제로 1회 이상 분석을 완료한 사용자 수

2. **1주 재방문률**

   * 정의: 첫 분석 완료 후 7일 이내 1회 이상 재방문한 사용자 비율

3. **사용자당 평균 등록 영양제 수**

   * 정의: 분석 완료 사용자 1인당 등록한 영양제 평균 개수

### 12-2. 추가 확인 필요

* KPI **목표 수치**는 아직 미정

  * 예: 첫 달 분석 완료자 300명
  * 예: 1주 재방문률 25%
  * 예: 사용자당 평균 등록 영양제 수 4개

---

## 13. 리스크 및 가정

### 13-1. 주요 리스크

1. **의료 서비스 오인 리스크**

   * 감기, 부족 성분, 조합 추천 표현이 진단/치료로 오해될 수 있음

2. **입력 복잡도 리스크**

   * 제품명, 성분, 함량, 복용량 입력이 많아 초기 이탈 가능성이 있음

3. **AI 정확도 리스크**

   * OCR/AI 추출 결과가 부정확할 수 있음

4. **기준 데이터 리스크**

   * 국내 기준 영양성분 기준의 출처와 범위가 확정되지 않으면 분석 일관성이 흔들릴 수 있음

5. **운영비 리스크**

   * 무료 서비스에서 OCR/설명용 AI API를 함께 쓰면 비용 증가 가능성이 있음

6. **설명 신뢰도 리스크**

   * 근거를 너무 적게 보여주면 신뢰도가 낮아지고, 너무 많이 보여주면 화면이 복잡해질 수 있음

### 13-2. 가정

1. 초기 MVP에서는 정식 관리자 페이지 없이 운영 가능하다.
2. 국내 기준 영양성분 기준 데이터는 확보 가능하다.
3. OCR 실패 시 직접 입력 전환으로 사용자 흐름을 유지할 수 있다.
4. 부족 가능 성분 추천은 초기에는 일반 가이드 수준으로 제공한다.
5. 생성형 AI는 판정이 아니라 설명 보조 역할로 제한한다.

## 이번 단계 확정 사항

1. Well Track의 1단계 PRD를 최종 확정본 형태로 정리함
2. 결과 설명 수준은 **전문가 수준이 아니라 생성형 AI 답변 수준의 쉬운 설명**으로 확정
3. 근거출처 범위는 아래처럼 확정

   * A: 사용자 입력 기반 근거
   * B: 국내 기준 영양성분 기준 및 내부 룰 기반 근거
   * C: 일반 건강 가이드 기반 설명
   * D: 생성형 AI 요약 설명
4. 표현 제한은 아래로 확정

   * “감기 치료”, “질환 개선 보장”, “의학적 처방” 등은 제외
   * “면역/감기 대비 목적의 일반 가이드” 수준으로 제한
5. KPI는 아래 3개로 확정

   * 첫 달 분석 완료자 수
   * 1주 재방문률
   * 사용자당 평균 등록 영양제 수
