# 2단계 상세 기능 명세서 최종 확정본

# 0. 공통 기준

## 0-1. 공통 원칙

* 본 문서는 **개발 티켓으로 바로 분해 가능한 수준**의 기능 명세를 목표로 한다.
* 기능 우선순위는 `P0 / P1 / P2`로 유지한다.
* 사용자 기능과 운영 기능을 구분한다.
* **운영 기능은 정식 관리자 UI를 전제로 하지 않는다.**
* 운영 대응은 **내부 운영 절차 / 내부 API / 로그 기반 대응**을 기본으로 한다.
* 본 단계에서는 **예외 상황 분류와 처리 원칙**만 정의한다.
* **HTTP 상태, 재시도 여부, 사용자 메시지, 공통 에러 코드 문자열, 운영 대응 상세 정책은 7단계에서 통일**한다.

## 0-2. 확정 정책 반영

* 기준 영양성분 데이터 출처: **한국인 영양섭취기준(KDRIs)**
* 관리자 UI: **없음**
* 비밀번호 정책:

  * 8~16자
  * 영문 대문자 1개 이상
  * 영문 소문자 포함
  * 숫자 포함
  * 특수문자 포함
* 업로드 허용 확장자:

  * `jpg`, `jpeg`, `png`, `webp`, `heic`
  * **가정**: 향후 모바일 원본 확장자가 추가 확인되면 운영 허용 목록에 추가
* 업로드 최대 용량:

  * **30MB**
* 분석 중 영양제 수정:

  * **불가**
* 분석 후 영양제 수정:

  * **허용**
* 분석 실행 전:

  * **분석 전 확인 단계 필수**
* 동일 분석 요청:

  * **처리 중/완료 모두 재사용**
* 재사용 결과:

  * **배지 + 안내 문구 표시**
* 문의 접수:

  * **비로그인 불가**
* 민감정보 확장 수집:

  * MVP 범위 밖
  * 향후 도입 시 **별도 동의 필요**

## F-001 회원가입

* **우선순위**: P0
* **기능명**: 이메일 회원가입
* **기능 설명**: 이메일, 비밀번호, 이름으로 계정을 생성한다.
* **목적**: 사용자별 영양제/분석 기록을 누적 저장하기 위함
* **사용자 유형**: 비로그인
* **사전 조건**:

  * 동일 이메일 계정이 없어야 함
  * 약관/개인정보 동의 가능 상태여야 함
* **입력값**:

  * `email`: string, 필수, 이메일 형식, 최대 255자, 중복 불가
  * `password`: string, 필수, 8~16자, 대문자 1개 이상 + 소문자 + 숫자 + 특수문자 포함
  * `name`: string, 필수, 1~30자
  * `agree_terms`: boolean, 필수, true
  * `agree_privacy`: boolean, 필수, true
* **처리 로직**:

  1. 입력값 검증
  2. 이메일 중복 확인
  3. 비밀번호 해시 처리
  4. `users` 생성
  5. 기본 상태 `ACTIVE`
* **출력값**:

  * `user_id`
  * `email`
  * `name`
  * `user_status`
  * `created_at`
* **상태값(enum 포함)**:

  * `user_status`: `ACTIVE | SUSPENDED | DELETED`
* **예외 처리**:

  * 이메일 중복
  * 비밀번호 정책 불충족
  * 약관/개인정보 미동의
  * DB 저장 실패
* **권한 체크**:

  * 로그인 사용자는 접근 불가
* **로그/이벤트 포인트**:

  * `auth.signup.requested`
  * `auth.signup.succeeded`
  * `auth.signup.failed`
* **관련 API**:

  * `POST /auth/signup`
* **관련 DB 테이블**:

  * `users`

* **목적**: 보호된 사용자 데이터 접근 허용

* **사전 조건**:

  * `email`: string, 필수, 이메일 형식
  * `password`: string, 필수
* **처리 로직**:

  1. 이메일 조회
  2. 비밀번호 비교
  3. 계정 상태 확인
  4. 세션/토큰 발급
  5. 마지막 로그인 시간 갱신
* **출력값**:

  * `session_status`
* **예외 처리**:

  * 계정 없음

  * 정지/삭제 계정
  * 세션 저장 실패
* **권한 체크**:

  * 로그인 사용자는 접근 불가
* **로그/이벤트 포인트**:

  * `auth.login.requested`
  * `auth.login.succeeded`
  * `auth.login.failed`
* **관련 API**:

  * `POST /auth/login`
* **관련 DB 테이블**:

  * `users`
  * `user_sessions`

## F-010 영양제 이미지 업로드

* **우선순위**: P0
* **기능명**: 영양제 성분표 이미지 업로드
* **기능 설명**: 사용자가 영양제 성분표 사진을 업로드하여 OCR 분석 준비 상태를 만든다.
* **목적**: 직접 입력 부담 완화
* **사용자 유형**: 일반사용자
* **사전 조건**:

  * 로그인 상태
* **입력값**:

  * `image_file`: file, 필수

    * 허용 확장자: `jpg`, `jpeg`, `png`, `webp`, `heic`
    * 최대 용량: 30MB
  * `label`: string, 선택, 최대 100자
* **처리 로직**:

  1. 확장자 검증
  2. MIME 타입 검증
  3. 파일 시그니처 검증
  4. 스토리지 저장
  5. `supplements` 초안 생성
  6. `supplement_images` 생성
  7. OCR 요청 상태 생성
* **출력값**:

  * `supplement_id`
  * `image_id`
  * `supplement_status`
* **상태값(enum 포함)**:

  * `supplement_status`: `DRAFT | OCR_PENDING | OCR_DONE | CONFIRMED | ANALYZABLE | ANALYSIS_LOCKED | ARCHIVED`
* **예외 처리**:

  * 허용되지 않은 확장자
  * 용량 초과
  * 손상 파일
  * 업로드 실패
  * 스토리지 저장 실패
* **권한 체크**:

  * 로그인 사용자만 가능
* **로그/이벤트 포인트**:

  * `supplement.image_upload.requested`
  * `supplement.image_upload.succeeded`
  * `supplement.image_upload.failed`
* **관련 API**:

  * `POST /supplements/images`
* **관련 DB 테이블**:

  * `supplements`
  * `supplement_images`

## F-011 OCR 추출 결과 확인/수정 후 등록 확정

* **우선순위**: P0
* **기능명**: OCR 추출 결과 검수
* **기능 설명**: OCR 추출 결과를 사용자가 확인하고 수정한 뒤 영양제를 등록 확정한다.
* **목적**: OCR 오인식 보정
* **사용자 유형**: 일반사용자
* **사전 조건**:

  * 로그인 상태
  * 대상 영양제가 본인 소유
  * `supplement_status = OCR_DONE`
* **입력값**:

  * `supplement_id`: uuid, 필수
  * `product_name`: string, 필수, 1~100자
  * `source_type`: enum, 필수, `IMAGE | HYBRID`
  * `serving_basis_type`: enum, 필수, `PER_SERVING | PER_DAY`
  * `daily_serving_count`: decimal, 선택, 0 초과
  * `ingredient_list`: array, 필수, 1개 이상

    * `ingredient_name_raw`: string, 필수
    * `amount_value`: decimal, 필수, 0 초과
    * `amount_unit`: string, 필수
    * `is_primary_display_value`: boolean, 선택
* **처리 로직**:

  1. OCR 결과 조회
  2. 사용자 수정값 반영
  3. 필수 성분/함량/단위 검증
  4. `supplement_ingredients` 저장
  5. 상태를 `CONFIRMED` 또는 `ANALYZABLE`로 전환
* **출력값**:

  * `supplement_id`
  * `confirmed_ingredient_count`
  * `supplement_status`
* **상태값(enum 포함)**:

  * `ingredient_match_status`
  * `supplement_status`
* **예외 처리**:

  * 성분 없음
  * 함량/단위 누락
  * 본인 소유 아님
  * OCR 결과 없음
* **권한 체크**:

  * 본인 영양제만 가능
* **로그/이벤트 포인트**:

  * `supplement.ocr_review.saved`
  * `supplement.ocr_review.failed`
* **관련 API**:

  * `PUT /supplements/{supplement_id}/confirm`
* **관련 DB 테이블**:

  * `supplements`
  * `supplement_images`
  * `supplement_ingredients`

## F-013 등록 영양제 수정/보관 처리

* **우선순위**: P0
* **기능명**: 영양제 수정 및 보관 처리
* **기능 설명**: 사용자가 본인 영양제 정보를 수정하거나 분석 대상에서 제외한다.
* **목적**: 잘못된 입력값 정정 및 복용 목록 관리
* **사용자 유형**: 일반사용자
* **사전 조건**:

  * 로그인 상태
  * 본인 영양제 존재
  * 현재 `PENDING | PROCESSING` 상태 분석에 포함되어 있지 않아야 함
* **입력값**:

  * `supplement_id`: uuid, 필수
  * 수정 시:

    * `product_name`
    * `serving_basis_type`
    * `daily_serving_count`
    * `ingredient_list`
  * 보관 시:

    * `archive_reason`: string, 선택
* **처리 로직**:

  1. 소유권 확인
  2. 해당 영양제가 진행 중 분석에 포함되어 있는지 확인
  3. 진행 중 분석 포함 시 수정 차단
  4. 허용 시 데이터 수정 또는 `ARCHIVED` 처리
* **출력값**:

  * `supplement_id`
  * `supplement_status`
  * `updated_at`
* **상태값(enum 포함)**:

  * `supplement_status`: `CONFIRMED | ANALYZABLE | ANALYSIS_LOCKED | ARCHIVED`
* **예외 처리**:

  * 존재하지 않는 영양제
  * 본인 소유 아님
  * 분석 중 수정 시도
* **권한 체크**:

## F-020 분석 요청 생성

* **우선순위**: P0
* **기능명**: 분석 실행 요청
* **기능 설명**: 사용자가 확인한 영양제 목록 기준으로 분석을 시작하거나 기존 동일 결과를 재사용한다.
* **목적**: 중복/과다/시간/조합/목적 추천 결과 생성
* **사용자 유형**: 일반사용자
* **사전 조건**:

  * 로그인 상태
  * 분석 전 확인 완료
  * 선택 영양제가 모두 본인 소유
  * 대상 영양제가 모두 `ANALYZABLE`
* **입력값**:

  * `supplement_ids`: uuid array, 필수
  * `purpose_codes`: string array, 선택
  * `confirmed_preview_id`: uuid, 필수
* **처리 로직**:

  1. 입력 영양제/소유권 확인
  2. preview 확인 완료 여부 검증
  3. fingerprint 생성

     * 사용자 ID
     * supplement_id 목록
     * supplement version 또는 updated_at
     * purpose_codes
  4. 동일 fingerprint의 기존 `analysis_run` 조회
  5. 있으면 기존 결과 재사용
  6. 없으면 새 `analysis_run` 생성
  7. 분석 대상 영양제를 `ANALYSIS_LOCKED`로 잠금
* **출력값**:

  * `analysis_run_id`
  * `analysis_status`
  * `is_reused_result`
  * `reuse_reason`
* **상태값(enum 포함)**:

  * `analysis_status`
  * `reuse_reason`: `SAME_FINGERPRINT`
* **예외 처리**:

  * preview 미확인
  * 분석 가능 상태 아님
  * 본인 영양제 아님
  * fingerprint 계산 대상 누락
* **권한 체크**:

  * 본인 영양제만 분석 가능
* **로그/이벤트 포인트**:

  * `analysis.requested`
  * `analysis.reused`
  * `analysis.created`
* **관련 API**:

  * `POST /analyses`
* **관련 DB 테이블**:

  * `analysis_previews`
  * `analysis_runs`
  * `supplements`

# 7. 공통 예외처리 연결 규칙

> 2단계에서는 **예외 상황 종류와 처리 원칙**만 정의한다.
> HTTP 상태, 공통 에러 코드, 사용자 메시지, 재시도 정책 상세는 7단계에서 통일한다.

## 7-1. 모든 기능 공통

* 입력 검증 실패 고려
* 인증/권한 실패 고려
* 리소스 없음 고려
* 외부 의존성 실패 고려
* DB/스토리지 실패 고려
* 로그 기록 필수
* request 단위 추적 가능해야 함

## 7-2. 업로드 기능 공통

* 확장자 검증
* MIME 타입 검증
* 파일 시그니처 검증
* 파일 용량 30MB 제한
* 스토리지 실패 정리 작업 고려
* OCR 실패 fallback 고려

## 7-3. 분석 기능 공통

* 분석 전 확인 미완료 시 차단
* 분석 중 수정 차단
* 동일 분석 요청 재사용
* KDRIs 기준 데이터 누락 시 부분 분석 또는 제한 처리
* AI 설명 실패 시 기본 결과 fallback

## 7-4. 운영 기능 공통

* 정식 관리자 UI 없음
* 내부 운영 절차 / 내부 API / 로그 기반 대응
* 기준 데이터/룰 변경 이력 기록 필수

## 이번 단계 확정 사항

1. **2단계 상세 기능 명세서 최종 확정본**을 작성함
2. 사용자 기능과 운영 기능을 구분하고, 모든 기능에 ID를 부여함
3. 확정 정책을 반영함

   * KDRIs 기준
   * 관리자 UI 없음
   * 비밀번호 정책 확정
   * 업로드 확장자/용량 확정
   * 분석 전 확인 필수
   * 분석 중 수정 불가 / 분석 후 수정 허용
   * 동일 분석 재사용
   * 재사용 결과 배지/문구 표시
   * 비로그인 문의 불가
4. 공통 서버 예외처리는 **예외 상황 분류와 처리 원칙까지만** 2단계에 반영하고, 상세 응답 표준은 7단계로 넘김
5. 이후 단계에서 바로 사용할 수 있도록 기능 간 선후관계와 공통 테이블/enum 초안을 함께 정리함

## 추가 확인 필요 항목

1. 모바일 원본 추가 확장자 허용 목록을 언제 확정할지
2. 분석 결과 재사용 배지 문구를 최종 UI에서 어떻게 표현할지
3. `ANALYSIS_LOCKED`를 영양제 단위로 둘지, 분석 대상 조합 단위로 둘지
4. P1 부족 가능 성분 추천의 상세 노출 포맷
5. 내부 운영 API의 인증 방식과 접근 제한 방식

## 충돌 가능 항목

1. `ANALYSIS_LOCKED` 정책은 6단계 ERD/API 설계 시 잠금 방식이 명확히 정해지지 않으면 구현 충돌 가능성이 있음
2. 동일 분석 재사용 정책은 fingerprint 설계가 느슨하면 잘못된 재사용이 발생할 수 있음
3. KDRIs 기준 버전 관리가 늦어지면 F-022, M-003 설계에 영향이 갈 수 있음
4. 운영 UI가 없기 때문에 내부 API/운영 절차가 약하면 실제 운영 부담이 커질 수 있음
