아래는 지금까지 확정한 내용과 방금 반영한 항목들까지 모두 포함한 **6단계 최종 확정본**입니다.
이번 확정본은 마스터 프롬프트의 6단계 출력 형식을 따르며, 5단계에서 확정된 핵심 정책인 **비밀번호 재설정 이메일 링크 5분**, **영양제 soft delete**, **삭제 상세 읽기 전용**, **사용자/운영자 복구 가능**, **문의 첨부 없음**, **/app 카드형 대시보드**, **OCR/분석 상태 화면의 재시도 UX**를 그대로 반영했습니다. 또한 서버 예외처리 체크리스트의 공통 원칙, HTTP 상태 코드 확장, 로그/모니터링/운영 대응 기준도 함께 반영했습니다.   

# 충돌 항목

1. 기존 6단계 초안의 `auth_refresh_tokens` 기반 JWT 가정은 폐기하고, **서버 세션 기반 인증**으로 정정한다.
2. 따라서 인증 저장 구조는 `auth_refresh_tokens` 대신 `user_sessions` 중심으로 설계한다.
3. 공통 에러 코드의 기본 체계는 마스터 프롬프트 기준을 유지하되, 예외처리 체크리스트에 있는 `429/502/503/504`는 **6단계 정책 반영 + 7단계 최종 확정 대상**으로 둔다.

# 2. API 명세 초안

## 2-0. 공통 API 정책

### 공통 성공 응답

```json id="0dxawc"
{
  "success": true,
  "data": {},
  "meta": {}
}
```

### 공통 실패 응답

```json id="gz4svw"
{
  "success": false,
  "code": "INTERNAL_ERROR",
  "message": "일시적인 오류가 발생했습니다. 잠시 후 다시 시도해주세요.",
  "detail": {},
  "request_id": "trace-id"
}
```

### 공통 예외처리 원칙

* 모든 예외는 사용자 메시지, 로그 포인트, 운영 대응, 재시도 가능 여부를 함께 정의한다.
* 내부 구현 정보는 응답에 노출하지 않는다.
* 401과 403은 명확히 구분한다.
* 업로드 성공 전까지 DB 상태 확정 금지, DB 쓰기 실패 시 부분 성공 금지.
* 서버 예외처리는 전역 핸들러 + 공통 포맷 + `request_id` 기반 추적을 적용한다. 

### 공통 에러 코드 체계

* 기본: `INVALID_INPUT`, `UNAUTHORIZED`, `FORBIDDEN`, `NOT_FOUND`, `CONFLICT`, `INTERNAL_ERROR`
* 확장 후보: `TOO_MANY_REQUESTS`, `BAD_GATEWAY`, `SERVICE_UNAVAILABLE`, `GATEWAY_TIMEOUT`
* 최종 코드는 7단계에서 확정한다.

## 2-1. Auth

### A-001 회원가입

* Method: `POST`
* Endpoint: `/api/v1/auth/signup`
* 설명: 사용자 회원가입
* 인증 필요 여부: 아니오
* 권한 필요 여부: 없음
* 요청 파라미터: 없음
* 요청 바디

  * `name string required max 100`
  * `email string required email`
  * `password string required min 8`
  * `consent_terms boolean required true`
  * `consent_privacy boolean required true`
* 응답 데이터

  * `user.id`, `user.email`, `user.name`, `user.role`
* 예외 상황

  * 이메일 형식 오류
  * 약관 미동의
  * 이메일 중복
  * DB 저장 실패
* 에러 코드

  * `INVALID_INPUT`, `CONFLICT`, `INTERNAL_ERROR`
* 재시도 가능 여부: 일부 가능
* 로그 포인트

  * 성공/실패, 이메일 해시, request_id

### A-002 로그인

* Method: `POST`
* Endpoint: `/api/v1/auth/login`
* 설명: 이메일/비밀번호 로그인
* 인증 필요 여부: 아니오
* 권한 필요 여부: 없음
* 요청 바디

  * `email string required`
  * `password string required`
* 응답 데이터

  * `user.id`, `user.email`, `user.name`, `user.role`
  * 세션 쿠키 설정
* 예외 상황

  * 인증 실패
  * 정지/탈퇴 대기 계정
  * 세션 저장소 장애
  * 로그인 시도 초과
* 에러 코드

  * `INVALID_INPUT`, `UNAUTHORIZED`, `FORBIDDEN`, `TOO_MANY_REQUESTS`, `SERVICE_UNAVAILABLE`, `INTERNAL_ERROR`
* 재시도 가능 여부: 조건부 가능
* 로그 포인트

  * 로그인 실패 횟수, 이메일 해시, IP 해시, request_id

### A-003 로그아웃

* Method: `POST`
* Endpoint: `/api/v1/auth/logout`
* 설명: 현재 세션 로그아웃
* 인증 필요 여부: 예
* 권한 필요 여부: 로그인 사용자
* 요청 바디: 없음
* 응답 데이터

  * `logged_out boolean`
* 예외 상황

  * 세션 없음
  * 세션 저장소 장애
* 에러 코드

  * `UNAUTHORIZED`, `SERVICE_UNAVAILABLE`, `INTERNAL_ERROR`
* 재시도 가능 여부: 가능
* 로그 포인트

  * session revoke, request_id

### A-004 내 정보 조회

* Method: `GET`
* Endpoint: `/api/v1/auth/me`
* 설명: 현재 로그인 사용자 정보
* 인증 필요 여부: 예
* 권한 필요 여부: 로그인 사용자
* 요청 파라미터: 없음
* 응답 데이터

  * `id`, `email`, `name`, `role`, `status`

# 3. 필수 보안 정책

## 3-1. 인증 정보 저장 방식

* 인증은 **서버 세션 기반**으로 확정한다.
* 세션 식별자는 `HttpOnly + Secure + SameSite` 쿠키로 저장한다.
* 세션 저장소는 **가정: Redis 우선**으로 두되, 7단계에서 최종 확정한다.
* 세션 저장소 장애 시 **fallback 없이 완전 차단**한다.

  * 로그인 불가
  * 인증 필요 API 전부 `503`
  * 공개/정적 페이지 외 보호 자원 접근 금지

## 3-2. 비밀번호 암호화 정책

* 비밀번호는 `Argon2id` 우선
* 대안은 `bcrypt cost 12+`
* 평문 저장, 복호화 가능한 저장 금지
* 비밀번호 변경 시 모든 세션 강제 만료

## 3-3. 세션 처리 방식

* 로그인 시 세션 생성
* 로그아웃 시 현재 세션 폐기
* 탈퇴/사용자 차단/즉시 삭제 시 해당 사용자의 모든 세션 폐기
* 세션 만료 정책은 7단계에서 구체 수치 확정
* 세션 유실/저장소 장애 시 재로그인 유도

## 3-4. 접근 제어 정책

* 기본 owner check 적용
* 운영자만 `/api/v1/admin/*` 접근 가능
* 삭제된 영양제는 목록 숨김, 상세 읽기 전용, 복구만 허용
* 사용자 화면에는 운영자 내부 메모 절대 노출 금지
* 즉시 hard delete는 운영자만 가능

## 3-5. 입력값 검증 정책

* 서버에서 최종 검증
* 이메일 형식, 문자열 길이, UUID, enum whitelist 검증
* 정렬 필드 whitelist 검증
* `memo`, `content`, `dosage_info`는 trim 후 저장
* 클라이언트 검증 실패와 서버 검증 실패 메시지를 분리

## 3-6. CORS 정책

* 허용 origin whitelist만 허용
* `credentials=true`
* wildcard origin 금지
* 허용 메서드/헤더 최소화

## 3-7. CSRF / XSS / SQL Injection 대응

* CSRF: 쿠키 인증 사용 시 SameSite + Origin/Referer 검증 + CSRF token 검토
* XSS: 사용자 입력을 HTML로 직접 렌더링 금지, escape 적용
* SQL Injection: ORM/Query Builder parameter binding만 허용, raw query 최소화

## 3-8. 파일 업로드 보안 정책

* 허용 형식: `jpg`, `jpeg`, `png`, `webp`
* 최대 크기: **가정 10MB**
* MIME type + 확장자 + magic number 3중 검증
* 업로드 성공 전까지 DB 상태 확정 금지
* 실패 시 임시 파일 정리
* OCR 원본은 검수 완료 즉시 삭제
* 악성 파일 검사는 7단계 이후 스토리지/인프라와 함께 확정

## 3-9. 개인정보 최소 수집 원칙 기술 반영

* 회원가입: 이름, 이메일, 비밀번호만 수집
* 문의는 첨부 미지원 유지
* 로그에는 이메일 원문보다 해시 우선
* 영양제 메모/문의 내용에는 민감정보 입력 자제 문구 노출 권장

## 3-10. 개인정보 보관 및 삭제 정책 기술 반영

* 탈퇴 즉시 로그인 차단
* `users.status = DELETED_PENDING`
* 30일 유예 후 개인정보 hard delete
* `analyses`, `inquiries`는 익명화 후 보존
* 운영자는 예외적으로 즉시 hard delete 가능
* 즉시 hard delete는 관리자 권한, 사유, 액션 로그 필수
* `password_reset_tokens`는 만료 후 정리 배치
* `user_sessions`는 만료/폐기 후 정리 배치

## 3-11. 로그에 남기면 안 되는 민감 정보

* 비밀번호
* 세션 쿠키 값
* 재설정 토큰 원문
* 카드 정보
* 주민번호 등 민감정보
* 파일 원문 데이터
* OCR 원본 공개 URL

## 3-12. 로그/모니터링 정책

* 반드시 남길 항목

  * `request_id`
  * `user_id` 또는 비회원 식별값
  * `endpoint`
  * `method`
  * `status_code`
  * `error_code`
  * 발생 시각
  * 처리 시간
  * 외부 API 호출 여부
* 최소 모니터링

  * 5xx 에러율
  * 응답시간
  * DB 연결 실패
  * 외부 API 장애
  * 인증 실패 급증
  * 배포 직후 오류 급증
* 운영자 액션 로그 필수 대상

  * 즉시 hard delete
  * 문의 상태 변경
  * 내부 메모 작성/수정
  * 사용자 차단/세션 강제 무효화
* 위 원칙은 예외처리 체크리스트의 로그/모니터링 요구를 반영한다. 

## 3-13. Rate Limit 정책

* 정책 적용 대상

  * 로그인
  * 비밀번호 재설정 요청
  * OCR 업로드
  * 분석 실행
  * 문의 등록
* 초과 시 `429 TOO_MANY_REQUESTS`
* 구체 수치는 7단계 또는 8단계에서 최종 확정
* **가정(초기 운영안)**

  * 로그인: 이메일 기준 5회/10분, IP 기준 20회/10분
  * 비밀번호 재설정 요청: 이메일 기준 3회/30분, IP 기준 10회/30분
  * OCR 업로드: 사용자 기준 10회/10분
  * 분석 실행: 사용자 기준 5회/10분
  * 문의 등록: 사용자 기준 5회/30분
* 숫자는 추후 조정 가능하고, 현재는 구현 준비용 초기안으로만 둔다. 

## 3-14. 외부 연동 장애 정책

* 외부 이메일 발송은 timeout, retry, backoff 정의
* `502`: 외부 응답 오류
* `504`: 외부 timeout
* `503`: 일시적 서비스 불가
* 무한 재시도 금지
* 내부 DB 오류와 외부 연동 장애를 로그에서 구분

## 3-15. 운영 환경 비밀키 관리 원칙

* 운영 비밀키는 Secret Manager/플랫폼 비밀변수 사용
* `.env`는 로컬 전용
* Git 커밋 금지
* 환경별 분리(dev/staging/prod)
* 키 회전 정책 수립

# 4. 개발 시 주의사항

1. **상태 API와 결과 API를 분리 유지**

   * OCR과 분석은 비동기 흐름이므로 상태 조회와 결과 조회를 분리해야 예외처리가 단순해진다. 이 구조는 5단계의 OCR 상태 화면, 분석 진행 상태 화면과도 일치한다. 

2. **snapshot 구조를 반드시 유지**

   * 삭제되거나 수정된 영양제가 과거 분석 결과를 훼손하지 않도록 `analysis_targets` 스냅샷을 유지해야 한다.

3. **원본 파일 삭제 시점 주의**

   * OCR 검수 전에는 원본이 필요하므로 검수 완료 전 삭제 금지.
   * 검수 완료 즉시 삭제 원칙과 UI 플로우가 충돌하지 않도록 백엔드 상태 전이를 맞춘다. 

4. **부분 성공 금지**

   * 업로드 성공 전 DB 확정 금지
   * 트랜잭션 실패 시 중간 상태 방치 금지
   * hard delete 중 일부만 익명화되는 상태 금지

5. **운영자 액션은 반드시 감사 가능해야 함**

   * 즉시 hard delete, 문의 상태 변경, 내부 메모는 모두 `admin_action_logs` 기록 필수

6. **세션 저장소 장애는 P0 장애로 취급**

   * fallback 없이 완전 차단 정책이므로 운영 알림과 상태페이지 대응이 필요

7. **결과 JSON은 외부 DTO와 내부 저장을 분리해서 생각**

   * 7단계에서 API DTO 고정
   * 내부 저장은 `result_version`으로 버전 관리

8. **공통 에러 코드는 7단계에서 최종 확정**

   * 6단계에서는 확장 코드를 정책에 반영했지만, 실제 구현 전 프로젝트 공통 코드 테이블을 7단계에서 닫아야 한다. 

9. **사용자 메시지는 일시적 장애와 영구 실패를 구분**

   * “잠시 후 다시 시도해주세요”
   * “접근 권한이 없습니다”
   * “로그인 상태가 만료되었습니다”
   * “지원하지 않는 파일 형식입니다”
   * 이 메시지 분리는 예외처리 체크리스트의 UI 기준과 맞춘다.

# 이번 단계 확정 사항

1. 인증 방식은 **서버 세션 기반**으로 확정했다.
2. `auth_refresh_tokens` 대신 `user_sessions` 구조로 정정했다.
3. `admin_action_logs`를 **P0 ERD 엔터티**로 포함했다.
4. OCR 원본 파일은 **검수 완료 후 즉시 삭제**, 실패 건은 재시도/직접 입력 전환 후 삭제로 확정했다.
5. 계정 탈퇴는 **즉시 로그인 차단 + 30일 유예 후 개인정보 hard delete + 분석/문의 익명화 보존**으로 확정했다.
6. 운영자는 예외적으로 **즉시 hard delete**가 가능하되, 관리자 권한/사유/액션 로그를 필수로 하도록 확정했다.
7. 운영자 전용 문의 관리 화면 범위를 문의 목록, 상태 변경, 내부 메모, 관련 분석/영양제 링크 확인으로 확정했다.
8. 분석 재사용은 **완전 일치 조건**에서만 허용하도록 확정했다.
9. 결과 JSON은 **7단계에서 API DTO 고정**, 내부 저장은 `result_version`으로 버전 관리하도록 확정했다.
10. 서버 예외처리 정책으로 `request_id`, 전역 예외 핸들러, 공통 실패 응답, 401/403 구분, 429/502/503/504 확장 후보, 로그/모니터링/운영 대응 원칙을 추가했다.

# 충돌 가능 항목

1. 7단계에서 세션 저장소를 DB로 선택하면 `user_sessions` 운용 방식과 장애 대응 문구가 일부 조정될 수 있다.
2. 7단계에서 공통 에러 코드를 단순화하면 `502/503/504`는 외부 응답 코드가 아니라 내부 로그 구분용으로만 축소될 수 있다.
3. 8단계에서 비동기 큐 구조가 도입되면 OCR/분석 재시도 정책과 상태 enum이 더 세분화될 수 있다.

# 바로 다음 단계 진행 가능 항목

* 7단계 기술 스택 및 아키텍처 확정
* 세션 저장소/Redis 여부 확정
* 공통 에러 코드 최종표 확정
* 결과 API DTO 스키마 초안 확정
* Rate Limit 수치 최종 확정
