아래는 사용자가 추가로 확정한 내용을 반영해 **0단계만 다시 정리한 최종본**입니다. 기준 문서와 이번 대화 입력을 함께 반영했습니다. 

## 충돌 항목

1. **“감기 같은 질병”을 사용자 목적에 포함**

   * 사용자의 요청에 따라 포함 가능함.
   * 다만 **건강관리 목적 추천 서비스**와 **질병 관련 추천**은 경계가 흐려질 수 있음.
   * 따라서 MVP에서는 아래처럼 제한하는 것이 적절함.

     * 허용: “피로”, “면역/감기 대비”, “수면”, “운동”, “눈 건강” 같은 **일반 목적/상태 중심 표현**
     * 비허용: 진단, 처방, 치료 대체처럼 보이는 표현
   * 즉, **“감기 치료”가 아니라 “면역/감기 대비 목적” 수준으로 제한** 필요

2. **“전문가 수준 제공”과 법적/서비스 책임 범위**

   * 사용자 요청상 설명 품질은 높게 가져갈 수 있음.
   * 하지만 서비스 성격상 **의료 진단/처방으로 오해되지 않도록 고지 문구**가 필요함.
   * 따라서 “전문가 수준의 설명”은 가능하되, **의학적 진단 확정 표현은 제외**해야 함.

3. **입력 UI에서 체크박스 사용 방식**

   * “1일 섭취량 / 1회분 섭취량 둘 다 할 수 있게 체크박스”
   * “함량도 체크박스로 여러 개 중 하나 선택”
   * 여기서 **여러 개 중 하나 선택**은 일반적으로 체크박스보다 **라디오 버튼/단일 선택 UI**가 더 적절함.
   * 현재 단계에서는 사용자의 의도를 존중해 **‘선택형 입력’으로 확정**하고, 구체 UI 컴포넌트는 다음 단계에서 정리하는 것이 적절함.

# 1. 서비스 요약

## 1-2. 서비스 정의

Well Track는 사용자가 현재 복용 중인 여러 영양제를 **제품명, 성분, 함량, 복용 기준량** 단위로 등록하면, 이를 성분 기준으로 통합 분석하여 아래 정보를 제공하는 웹서비스다.

* 중복 성분
* 과다 복용 가능성
* 부족 가능 성분
* 복용 시간 추천
* 같이 먹으면 좋은 조합
* 같이 먹을 때 주의할 조합
* 목적 기반 추천

  * 예: 피로, 운동, 수면, **면역/감기 대비**

## 1-3. 구현/설계에 직접 영향을 주는 핵심 결정 사항

1. **이미지 업로드 + 직접 입력을 모두 지원**
2. **분석 결과를 일회성으로 끝내지 않고 사용자 계정 기준으로 누적 저장**
3. **국내 기준 영양성분 기준값을 우선 적용**
4. **복용 시간 추천은 룰 기반 우선**
5. **AI 설명 기반 해설을 붙일 수 있도록 확장 가능 구조 필요**
6. **부족 가능 성분 추천은 식단/생활습관 입력이 없어도 제공**

   * 단, 반드시 “식단/생활습관 정보 미입력 기준” 문구 표시
7. **입력 단위가 복잡함**

   * 제품명
   * 성분
   * 함량
   * 1회분 섭취량
   * 1일 섭취량
   * 기준 선택형 입력 필요
8. **무료 서비스이지만 AI API 2종류 사용 가능성 있음**

   * 비용 통제 구조 필요

# 3. MVP 핵심 범위

## 3-1. MVP 범위 축소 제안

현재 아이디어는 범위가 넓기 때문에, MVP는 아래처럼 줄이는 것이 적절합니다.

### MVP 핵심 원칙

* **분석 정확도와 입력 안정성 확보가 우선**
* **진단형 서비스가 아니라 복용 관리/분석 서비스로 제한**
* **의료적 표현은 피하고, 정보 제공형 가이드로 설계**
* **AI는 설명 보조 역할**, 핵심 판정은 룰 기반 우선

## 3-2. 기능 우선순위

### P0

1. 회원가입 / 로그인
2. 영양제 등록

   * 이미지 업로드
   * 직접 입력
3. 영양제 입력 항목 관리

   * 제품명
   * 성분명
   * 함량
   * 1회분 섭취량
   * 1일 섭취량
   * 기준 선택형 입력
4. OCR/AI 추출 결과 사용자 확인 및 수정
5. 성분명/함량 표준화
6. 여러 영양제의 성분 합산 분석
7. 중복 성분 표시
8. 과다 복용 가능성 표시
9. 복용 시간 추천

   * 룰 기반
10. 같이 먹으면 좋은 조합 / 주의 조합의 **기본 가이드**
11. 사용자 목적 기반 추천

* 예: 피로, 운동, 수면, 눈 건강, **면역/감기 대비**

12. 분석 결과 누적 저장 및 조회
13. 결과 화면에서 “줄이기/유지/검토 필요” 수준의 요약 가이드 제공

### P1

1. 부족 가능 성분 추천

   * 식단/생활습관 입력이 없더라도 제공
   * 단, **“식단/생활습관 미입력 기준의 일반 가이드” 문구 표시 필수**
2. 식단/생활습관 입력 기능 추가
3. 부족 가능 성분 추천 정확도 고도화
4. AI 설명 기반 해설 추가

   * 왜 중복인지
   * 왜 시간 추천이 나왔는지
   * 왜 조합 주의가 필요한지
5. 결과 설명의 자연어 요약 강화

### P2

1. 건강 상태/연령/성별 기반 개인화
2. 리마인더/복용 알림
3. 추천 제품 제안
4. 약물과의 상호작용
5. 관리자용 내부 운영 도구
6. 고도화된 전문 상담 연결

# 4. 주요 제약사항

1. **국내 기준 데이터 확보 필요**

   * 국내 기준값을 우선 적용하기로 했으므로, 기준 영양성분 데이터 소스가 필요함
   * 이 기준이 없으면 과다/부족 분석 정확도가 흔들릴 수 있음

2. **부족 가능 성분 추천 정확도 한계**

   * 식단/생활습관 정보 없이도 제공은 가능하나, 정확도는 제한적임
   * 따라서 안내 문구가 반드시 필요함

3. **의료적 해석 범위 제한 필요**

   * 감기 같은 표현을 넣더라도 진단/치료 서비스처럼 보이면 안 됨
   * “면역/감기 대비 목적” 수준으로 제한하는 정책 필요

4. **입력 구조가 복잡함**

   * 제품명/성분/함량/1회분/1일분 모두 입력하게 되면 입력 피로도가 높아질 수 있음
   * 이미지 업로드 후 자동 추출 + 사용자 수정 구조가 중요함

5. **AI 비용 및 안정성**

   * OCR용 AI API와 설명용 AI API를 둘 다 쓰면 비용과 응답속도 이슈가 생길 수 있음
   * 핵심 분석은 룰 기반으로 유지해야 운영 리스크가 낮음

6. **무료 서비스의 운영 범위 제한**

   * 무료 모델에서는 API 호출량, 저장 용량, 이미지 처리량 제한이 필요할 수 있음

7. **전문가 수준 설명 요청**

   * 설명 품질은 높게 설계할 수 있으나, 실제 법적 책임 범위와 분리된 안내 문구 필요

# 5. 추가 확인 필요 항목

1. **국내 기준 데이터의 실제 출처**

   * 현재는 “국내 기준”만 확정됨
   * 어느 기관/기준표를 따를지는 추가 확인 필요

2. **목적 기반 추천의 허용 표현 범위**

   * “감기”를 정확히 어떤 문구로 서비스에 노출할지 필요
   * 권장: “면역/감기 대비”
   * 추가 확인 필요: “감기 예방”, “감기 회복” 같은 표현 허용 여부

3. **입력 UI 방식 상세**

   * 1회분 / 1일 섭취량 선택 방식
   * 함량 단위 선택 방식
   * 현재는 “선택형 입력”까지만 확정

4. **AI API 2종의 역할 분리**

   * OCR/텍스트 추출용
   * 설명/해설 생성용
   * 구체적인 사용 위치는 다음 단계에서 확정 필요

5. **목표 KPI 수치**

   * 현재는 “분석 기능 제공 중심”까지만 확정
   * 수치 목표는 미정

6. **문의/오류 신고 대응 방식**

   * 관리자 페이지는 불필요하나, 최소한의 운영 대응 수단은 필요한지 확인 필요

## 충돌 가능 항목

1. “감기” 같은 질병 관련 표현은 진단/치료 서비스처럼 해석될 수 있음
2. 전문가 수준 설명을 강조할수록 법적 책임 범위와 충돌 가능성이 커짐
3. 입력 항목이 많아질수록 초기 사용자 이탈 가능성이 커짐
4. 무료 서비스인데 AI API 2종을 사용하면 운영비 부담이 커질 수 있음
