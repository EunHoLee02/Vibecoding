충돌 항목
1. 영양제 핵심 데이터 모델이 서로 다릅니다.
2단계 P0는 영양제 등록/검수 시 ingredient_list, serving_basis_type, daily_serving_count 같은
분석용 구조를 요구합니다. 반면 6단계 최종 API와 ERD 쪽은 직접 등록/OCR 확정 API가 name,
manufacturer, dosage_info, memo 중심으로만 보이고, 최종 DB 엔터티도 supplements,
supplement_ocr_jobs 중심으로 정리되어 있어 분석에 필요한 정규화된 성분 입력 구조가 빠져 있습니
다. 이 상태로는 영양제 CRUD, OCR 검수 저장, 분석 전 확인, 분석 실행을 일관되게 구현할 수 없습니
다.   
2. OCR 업로드 계약이 서로 다릅니다.
2단계 F-010은 POST /supplements/images와 supplement_images 기반 흐름을 전제로 하고, 업로드 직
후 supplements 초안 생성까지 포함합니다. 그런데 6단계 API는 POST /api/v1/supplements/ocr-jobs
멀티파트 업로드로 바뀌었고, 7~8단계 확정본은 다시 R2 presigned upload + OCR 원본 즉시 삭제 아
키텍처를 기준으로 잡았습니다. 즉, “백엔드가 직접 파일을 받는지”, “presigned URL만 발급하는지”, “초
안 supplement를 언제 만드는지”가 아직 하나로 닫히지 않았습니다. 
  
3. 인증 정책의 세부 규칙이 충돌합니다.

| DELETED 초안을 사용합니다. 반면 10a 최종 확정본은 비밀번호를 8~128 자  + 영문 / 숫자 / 특수문자로 바
꾸고 계정 상태를 active | disabled | pending_delete로 재정의했습니다. 6단계 ERD는 또 ACTIVE |
SUSPENDED | DELETED_PENDING | DELETED를 사용합니다. 최신 확정본 우선 원칙을 적용하더라도, 구현

괜찮을 수도 있고, 안 괜찮을 수도 있습니다.
경고 종류에 따라 다릅니다.
지금 제가 보여드린 코드는 구조 초안 중심이라서, 실제 파일이 아직 안 만들어졌거나, 아직 __init__.py,
경로, 심볼 정의가 안 맞으면 from ... import ... 부분에 경고가 뜰 수 있습니다. 이 경우는 자연스러운
임시 경고입니다.
다만 아래 경우면 나중에 실제 에러로 이어질 수 있어서 정리해야 합니다.
괜찮은 경우
1. 아직 파일을 안 만든 경우
예: from app.services.auth_service import AuthService
그런데 실제 auth_service.py를 아직 안 만들었으면 에디터 경고가 뜹니다.
이건 초안 단계에서는 괜찮습니다.
2. 아직 클래스/함수를 안 만든 경우
파일은 있는데 AuthService가 아직 없으면 경고가 뜹니다.
이것도 구현 전이면 괜찮습니다.
3. 순차 작성 중인 경우
제가 deps/auth.py를 먼저 보여주고, 그 안에서 참조하는 auth_service.py는 앞 단계 기준 존재
한다고 가정했기 때문에, 실제 프로젝트에 아직 반영 안 했으면 경고가 날 수 있습니다.
안 괜찮은 경우
1. 경로가 실제 구조와 다를 때

2. 순환 import
A가 B를 import하고, B가 다시 A를 import하면 경고가 아니라 실행 에러로 갈 수 있습니다.
3. 패키지 인식 문제

니다.

migration에서는 server_default=now()만 있고 DB trigger는 없습니다.
지금 구조는 ORM onupdate 기준입니다.
raw SQL update를 많이 쓰게 되면 별도 trigger가 필요할 수 있습니다.
