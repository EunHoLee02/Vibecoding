[이번 단계 추가 입력]
- 이번 단계 목표:
  1. P3에서 확정된 데이터 모델과 API 계약을 기준으로 백엔드 구현 지시문을 만든다.
  2. 백엔드는 프로토타입 시연에 필요한 최소 범위만 구현하도록 제한한다.
  3. endpoint, request body, response shape, 상태값, mock 처리 방식이 이후 단계에서 바뀌지 않도록 고정한다.
- 백엔드 구현 범위:
  1. health check 또는 root 확인용 최소 endpoint
  2. 영양제 직접 입력 처리 API
  3. 업로드/mock 처리 API
  4. 분석 실행 API
  5. 분석 결과 반환 API 또는 분석 실행 응답에 결과 포함
- 백엔드에서 제외할 범위:
  1. 로그인/회원가입
  2. 사용자별 세션/권한
  3. 관리자 기능
  4. 결제
  5. 문의/운영 API
  6. 외부 유료 API 필수 의존
  7. 복잡한 비동기 큐
- 분석 로직 요구:
  1. rule-based 우선
  2. 입력된 영양제 목록을 기반으로 중복 성분을 계산한다.
  3. 과다 가능성은 단순 기준표 또는 샘플 기준값으로 계산한다.
  4. 복용 시간 가이드는 고정 rule 또는 단순 매핑으로 계산한다.
  5. 결과는 항상 동일 입력에 동일 출력이 나오도록 결정적이어야 한다.
- 업로드/mock 처리 요구:
  1. 실제 OCR 필수 아님
  2. 파일 업로드를 받아도 내부에서는 샘플 파싱 결과를 반환해도 된다.
  3. 또는 업로드 없이 샘플 payload를 반환하는 stub API여도 된다.
  4. 단, 프론트엔드가 업로드 흐름을 시연할 수 있어야 한다.
- 저장 방식 요구:
  1. P4에서 확정한 방식만 사용한다.
  2. 필요 이상으로 DB/ORM을 확장하지 않는다.
  3. 프로토타입 시연 안정성을 우선한다.
- 코드 작성 방식 요구:
  1. 파일 단위로 작성
  2. 각 파일 역할 설명 포함
  3. 한 번에 너무 많은 파일을 만들지 말고 기능 묶음 단위로 작성
  4. 실제 실행 가능한 수준의 코드 초안이어야 한다.
- 테스트 요구:
  1. 최소 단위 테스트 케이스 목록 포함
  2. 실제 테스트 코드가 아니어도 되지만 endpoint별 검증 포인트는 적어야 한다.
- 에러 처리 요구:
  1. 성공/실패 응답 구조는 P3 계약을 따른다.
  2. 의료적 판단처럼 보이는 에러/응답 메시지 금지
  3. 파일 업로드 실패, 입력 검증 실패, 분석 실패 정도만 최소로 포함하면 된다.
