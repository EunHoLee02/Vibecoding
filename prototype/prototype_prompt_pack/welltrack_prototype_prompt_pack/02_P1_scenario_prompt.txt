[이번 단계 추가 입력]
- 이번 단계 목표:
  1. 사용자가 실제로 시연하게 될 핵심 사용자 시나리오를 고정한다.
  2. 프로토타입에서 어떤 흐름을 반드시 구현해야 하는지 확정한다.
  3. 이후 화면 설계와 API 설계에서 흔들리지 않도록 사용자 행동 단위를 고정한다.
- 프로토타입 핵심 시연 시간 목표:
  1. 전체 흐름은 3분 이내에 끝나야 한다.
- 반드시 포함해야 하는 사용자 시나리오:
  1. 사용자가 직접 영양제 정보를 입력한다.
  2. 또는 샘플 이미지/샘플 데이터를 선택한다.
  3. 분석 실행 버튼을 누른다.
  4. 결과 화면에서 중복 성분, 과다 가능성, 복용 시간 가이드를 확인한다.
- 반드시 포함해야 하는 결과 해석:
  1. 어떤 성분이 중복되는지
  2. 어떤 성분이 과다 가능성이 있는지
  3. 언제 먹는 것이 좋은지
  4. 전체 요약 한 줄
- 프로토타입에서 허용되는 단순화:
  1. 사용자는 익명 단일 사용자처럼 가정해도 된다.
  2. 데이터는 사용자 계정 단위로 저장하지 않아도 된다.
  3. OCR은 실제 업로드가 아니라 샘플 업로드/모의 처리로 대체 가능하다.
  4. 분석 결과는 고정 룰 또는 샘플 데이터 기반으로 반환 가능하다.
- 프로토타입에서 금지할 확장:
  1. 사용자별 기록 누적
  2. 관리자 개입 흐름
  3. 결제 흐름
  4. 문의/신고 흐름
  5. 의료적 진단/처방처럼 보이는 흐름
- 시연 우선순위:
  - P0:
    1. 입력
    2. 분석 실행
    3. 결과 표시
  - P1:
    1. 업로드 정확도 보강
    2. 결과 설명 보강
  - P2:
    1. 저장
    2. 개인화
    3. 운영 기능

[공통 강제 규칙]
- 로그인, 관리자, 결제, 문의, 운영 기능은 시나리오에 넣지 말아줘.
- 사용자 행동은 실제 화면/버튼/API로 이어질 수 있을 정도로 구체적으로 적어줘.
- 시나리오는 입력 -> 분석 실행 -> 결과 확인 흐름을 중심으로 고정해줘.
- 추천은 자동 승계하지 말고 확정만 다음 단계로 넘겨줘.
- Prototype Decision Register와 P0 authoritative_updates를 최우선으로 사용해줘.
- 결과 표현은 일반 건강 가이드 수준으로만 유지하고, 의료적 확정 표현은 금지해줘.
- 마지막에 반드시 아래 7개를 포함해줘.
  1. 이번 단계 확정 사항
  2. 다음 단계 입력으로 넘길 핵심 요약
  3. 추가 확인 필요 항목
  4. 충돌 가능 항목
  5. 바로 다음 단계 진행 가능 항목
  6. 단계 상태
  7. authoritative_updates YAML
