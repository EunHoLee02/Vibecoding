[이번 단계 추가 입력]
- 이번 단계 목표:
  1. P0~P6에서 확정된 범위, 화면, 데이터, API, 기술 구조를 최종 점검한다.
  2. 실제 코드 생성 전에 충돌, 누락, 과도한 확장 가능성을 정리한다.
  3. 이후 코드 생성 프롬프트에서 그대로 사용할 수 있는 handoff 문서를 만든다.
- 최종 점검 대상:
  1. 로그인/회원가입이 다시 들어가지 않았는지
  2. 관리자/결제/문의 기능이 다시 들어가지 않았는지
  3. P2 화면과 P6 프론트 구현 범위가 일치하는지
  4. P3 API 계약과 P5 백엔드 범위가 일치하는지
  5. P3 응답 구조와 P6 결과 화면 렌더링 항목이 일치하는지
  6. route 이름, endpoint 이름, 핵심 필드명이 단계 사이에서 바뀌지 않았는지
  7. rule-based 분석 원칙이 유지되는지
  8. OCR이 실제 연동이 아니라 stub/mock 우선 원칙을 유지하는지
- 최종 시연 흐름:
  1. 시작 화면 진입
  2. 직접 입력 또는 샘플 업로드/선택
  3. 분석 실행
  4. 결과 확인
  5. 다시 입력 또는 초기화
- 최종 코드 생성 전 고정하고 싶은 항목:
  1. 화면 목록
  2. route 목록
  3. endpoint 목록
  4. 핵심 데이터 모델
  5. 상태값(enum)
  6. 결과 응답 구조
  7. 백엔드/프론트엔드 구현 순서

[이번 단계 프롬프트]
P7단계 프로토타입 최종 점검 + 코드 생성 직전 handoff 문서 작성만 수행해줘.

반드시 아래 형식으로 정리해줘.
1. 최종 범위 점검 결과
2. 충돌 여부 점검 결과
3. 누락 여부 점검 결과
4. 최종 화면/route 고정 목록
5. 최종 API/endpoint 고정 목록
6. 최종 데이터 모델/상태값 고정 목록
7. 백엔드 handoff 요약
8. 프론트엔드 handoff 요약
9. 시연 체크리스트
10. 코드 생성 직전에 반드시 유지할 고정 규칙
11. 바로 다음 단계에서 사용할 최종 통합 요약

[공통 강제 규칙]
