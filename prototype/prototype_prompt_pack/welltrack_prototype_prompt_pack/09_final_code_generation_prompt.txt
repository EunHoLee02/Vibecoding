지금부터는 Well Track 프로토타입의 실제 코드 생성을 시작해줘.

중요:
- 이 작업은 정식 서비스 개발이 아니라 프로토타입 구현이다.
- 로그인/회원가입/권한/관리자/결제/문의/운영 기능은 구현하지 않는다.
- 범위를 임의로 확장하지 않는다.
- 이전에 확정된 route, endpoint, 필드명, 상태값, 응답 구조를 바꾸지 않는다.
- 프로토타입 핵심 흐름은 오직 입력 -> 분석 실행 -> 결과 확인이다.
- 분석은 rule-based 우선이다.
- OCR은 실제 연동이 아니라 stub/mock 우선이다.
- 결과 표현은 의료 진단/처방처럼 보이지 않게 일반 가이드 수준으로 유지한다.

[코드 생성 방식]
- 한 번에 전체 프로젝트를 모두 출력하지 말아줘.
- 아래 순서대로 기능 묶음 단위로 작성해줘.
  1. 전체 프로젝트 구조
  2. 백엔드 최소 실행 구조
  3. 백엔드 API 1차 구현
  4. 프론트엔드 최소 실행 구조
  5. 입력 화면 구현
  6. 결과 화면 구현
  7. 업로드/mock 흐름 연결
  8. 분석 실행 연결
  9. 실행 방법 정리
- 각 턴마다 이번에 생성한 파일 목록을 먼저 보여줘.
- 각 파일은 실제 실행 가능한 수준으로 작성해줘.
- 필요하면 mock 데이터나 stub 응답을 사용해도 된다.
- 과도한 추상화, 과도한 폴더 분리, 과도한 디자인 시스템 확장은 금지한다.

[이번 턴 작업 범위]
이번 턴에서는 먼저 아래만 수행해줘.
1. 전체 프로젝트 폴더 구조 제안
2. 백엔드 파일 구조
3. 프론트엔드 파일 구조
4. 실행에 필요한 최소 환경설정 파일 목록
5. 백엔드 구현 1차에 만들 파일 목록
6. 프론트엔드 구현 1차에 만들 파일 목록

[공통 강제 규칙]
- 확정되지 않은 새 기능 추가 금지
- 로그인, 관리자, 결제, 문의, 운영 기능 추가 금지
- endpoint 이름 변경 금지
- 필드명 변경 금지
- 상태값 변경 금지
- response shape 변경 금지
- 실제 OCR 필수화 금지
- 외부 유료 API 필수화 금지
- 코드가 길어지면 기능 묶음 단위로 끊어서 순서대로 작성해줘
- 마지막에는 반드시 아래를 포함해줘.
  1. 이번 턴에서 생성한 파일 목록
  2. 다음 턴에서 생성할 파일 목록
  3. 실행 순서 요약
  4. 충돌 가능 항목
