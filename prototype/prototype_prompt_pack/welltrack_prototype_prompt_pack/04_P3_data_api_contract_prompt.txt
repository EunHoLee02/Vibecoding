[이번 단계 추가 입력]
- 이번 단계 목표:
  1. 프로토타입에서 사용할 데이터 모델을 단순하고 일관되게 고정한다.
  2. 프론트엔드와 백엔드가 같은 필드명, 상태값, endpoint를 사용하도록 계약을 확정한다.
  3. 이후 단계에서 API 이름이나 응답 구조가 흔들리지 않게 최소 계약을 만든다.
- 프로토타입 데이터 모델 범위:
  1. 영양제 입력 데이터
  2. 업로드 또는 샘플 입력 데이터
  3. 분석 요청 데이터
  4. 분석 결과 데이터
- 프로토타입에서는 제외할 데이터 범위:
  1. 사용자 계정 데이터
  2. 관리자 데이터
  3. 결제 데이터
  4. 문의 데이터
  5. 운영 로그 데이터
- 입력 데이터에서 반드시 필요한 항목:
  1. product_name
  2. manufacturer optional
  3. ingredients
  4. each ingredient name
  5. each ingredient amount
  6. each ingredient unit
- 분석 결과에서 반드시 필요한 항목:
  1. supplements
  2. duplicated_ingredients
  3. risk_items
  4. timing_guides
  5. summary_text
- 업로드/OCR 처리 정책:
  1. 실제 OCR 필수 아님
  2. 업로드는 mock/stub 가능
  3. 프로토타입에서는 업로드 성공 후 샘플 파싱 결과를 반환해도 됨
- 상태값 설계 원칙:
  1. 최소 개수만 사용
  2. 프론트엔드와 백엔드에서 같은 문자열 사용
  3. 이후 단계에서 바꾸지 않도록 단순하게 정의
- endpoint 설계 원칙:
  1. 최소 개수
  2. 이름 단순화
  3. CRUD 전체보다 시연 흐름 중심
  4. 입력 -> 분석 -> 결과 흐름에 필요한 것만 포함
- 응답 포맷 원칙:
  1. 성공/실패 구조 통일
  2. result 화면에서 바로 렌더링 가능해야 함
  3. 중첩이 너무 깊지 않아야 함
- 기술 구현 제약:
  1. 프로토타입이므로 DB 저장은 선택 가능
  2. in-memory, sqlite, json mock 중 하나를 선택할 수 있음
  3. 단, 계약 필드명과 응답 형식은 이후 단계에서 유지해야 함

[이번 단계 프롬프트]
P3단계 프로토타입 데이터 모델 + API 계약 정의만 수행해줘.

반드시 아래 형식으로 정리해줘.
1. 프로토타입 핵심 데이터 모델 목록
2. 각 데이터 모델의 필드 정의
3. 필수 상태값(enum) 정의
4. 프로토타입 핵심 API 목록
5. 각 API의 method / endpoint / 목적
6. 각 API의 요청 body 또는 파라미터
7. 각 API의 응답 형식
8. 공통 성공/실패 응답 포맷
9. 다음 단계에서 바로 사용할 데이터/API 고정 요약

[공통 강제 규칙]
- 로그인, 사용자 계정, 관리자, 결제, 문의, 운영 로그 관련 모델/API는 넣지 말아줘.
- 데이터 모델은 시연에 필요한 최소 범위만 정의해줘.
- endpoint는 너무 많게 만들지 말고, 핵심 흐름에 필요한 최소 개수만 정의해줘.
- 필드명, enum, endpoint 이름은 이후 단계에서 그대로 유지할 수 있게 단순하고 일관되게 정해줘.
- 추천은 자동 승계하지 말고 확정만 다음 단계로 넘겨줘.
- Prototype Decision Register와 P2 authoritative_updates를 최우선으로 사용해줘.
- 결과 응답은 결과 화면에서 바로 렌더링 가능한 구조로 설계해줘.
- 의료적 진단/처방처럼 보이는 필드명이나 상태값은 금지해줘.
- 마지막에 반드시 아래 7개를 포함해줘.
  1. 이번 단계 확정 사항
  2. 다음 단계 입력으로 넘길 핵심 요약
  3. 추가 확인 필요 항목
  4. 충돌 가능 항목
  5. 바로 다음 단계 진행 가능 항목
  6. 단계 상태
  7. authoritative_updates YAML
