[이번 단계 추가 입력]
- 이번 단계 목표:
  1. 프로토타입 구현에 사용할 프론트엔드/백엔드 기술 조합을 단순하게 확정한다.
  2. 저장 방식과 mock/stub 처리 방식을 확정한다.
  3. 이후 P5, P6 단계에서 그대로 사용할 폴더 구조와 구현 순서를 고정한다.
- 기술 선택 원칙:
  1. 프로토타입은 빠르게 실행 가능해야 한다.
  2. 학습비용이 낮고 흔한 조합을 우선한다.
  3. 로그인, 권한, 복잡한 운영 인프라는 고려하지 않는다.
  4. 실제 서비스 확장성보다 시연 속도를 우선한다.
- 프론트엔드 요구:
  1. 입력 화면
  2. 업로드/샘플 선택 화면
  3. 결과 화면
  4. 간단한 로딩/오류 상태
- 백엔드 요구:
  1. 영양제 입력 처리 API
  2. 업로드/mock 처리 API
  3. 분석 실행 API
  4. 결과 반환 API
- 저장 방식 후보:
  1. in-memory
  2. sqlite
  3. json file mock
- 선택 원칙:
  1. 프로토타입 시연 안정성이 가장 중요하다.
  2. 여러 AI가 구현해도 가장 흔들리지 않는 선택을 우선한다.
  3. 너무 많은 선택지를 열어두지 말고 하나로 고정한다.
- OCR 처리 방식 후보:
  1. 실제 OCR 없음, 샘플 파싱 결과 반환
  2. 업로드 후 stub response 반환
- 분석 처리 방식:
  1. rule-based
  2. 외부 AI 의존 금지 또는 선택 제외
- 구현 구조 요구:
  1. 프론트엔드와 백엔드 폴더를 분리한다.
  2. 파일 수는 과도하게 많지 않게 한다.
  3. P5에서 백엔드 코드 작성 가능해야 한다.
  4. P6에서 프론트엔드 코드 작성 가능해야 한다.
- 시연 환경 요구:
  1. 로컬 실행 가능
  2. 복잡한 클라우드 설정 없이 실행 가능
  3. 환경변수는 최소화
- 제외할 기술 범위:
  1. Redis
  2. 메시지 큐
  3. 인증 세션 저장소
  4. 관리자 인프라
  5. 배포 자동화
  6. 외부 유료 API 필수 의존

[공통 강제 규칙]
- 로그인, 관리자, 결제, 문의, 운영 기능을 위한 기술은 포함하지 말아줘.
- 기술 선택은 빠른 프로토타입 제작 기준으로 하나의 최종안만 확정해줘.
- 후보를 여러 개 길게 비교하지 말고, 최종안 1개와 아주 짧은 이유만 제시해줘.
- 저장 방식은 하나만 확정해줘. 이후 단계에서 바꾸지 않게 해줘.
- OCR은 실제 연동보다 stub/mock 우선 원칙을 유지해줘.
- 외부 AI나 유료 API 의존을 필수로 만들지 말아줘.
- Prototype Decision Register와 P3 authoritative_updates를 최우선으로 사용해줘.
- 이후 단계에서 폴더명, route명, API 경로, 핵심 데이터 구조가 흔들리지 않도록 단순한 구조를 선택해줘.
- 마지막에 반드시 아래 7개를 포함해줘.
  1. 이번 단계 확정 사항
  2. 다음 단계 입력으로 넘길 핵심 요약
  3. 추가 확인 필요 항목
  4. 충돌 가능 항목
  5. 바로 다음 단계 진행 가능 항목
  6. 단계 상태
  7. authoritative_updates YAML
