[이번 단계 추가 입력]
- 이번 단계 목표:
  1. 프로토타입의 실제 화면 수를 고정한다.
  2. 각 화면의 목적, 입력 요소, 버튼, 이동 흐름을 확정한다.
  3. 이후 프론트엔드 구현과 API 계약에서 흔들리지 않도록 화면과 플로우를 단순하게 고정한다.
- 프로토타입에서 허용할 화면 수:
  1. 최소 3개
  2. 최대 5개
- 반드시 포함해야 하는 화면:
  1. 시작/입력 화면
  2. 업로드 또는 샘플 선택 화면
  3. 분석 결과 화면
- 선택 가능 화면:
  1. 분석 진행 상태 화면
  2. 샘플 선택 전용 화면
- 화면 설계 원칙:
  1. 로그인 관련 화면 금지
  2. 관리자 화면 금지
  3. 결제 화면 금지
  4. 문의/운영 화면 금지
  5. 시연 중 불필요한 분기 최소화
  6. 3분 이내 시연 가능한 흐름 유지
- 사용자 입력 방식:
  1. 직접 입력
  2. 샘플 업로드 또는 샘플 데이터 선택
- 결과 화면에서 반드시 보여줄 UI 정보:
  1. 입력한 영양제 목록
  2. 중복 성분 섹션
  3. 과다 가능성 섹션
  4. 복용 시간 가이드 섹션
  5. 요약 한 줄
- 프로토타입 UI 톤:
  1. 간단하고 직관적
  2. 복잡한 대시보드 형태 금지
  3. 핵심 기능 중심 단일 흐름 우선
- 반응형 범위:
  1. 데스크톱 우선
  2. 모바일에서도 깨지지 않는 정도면 충분
- 결과 화면 표현 제한:
  1. 의료 진단처럼 보이는 문구 금지
  2. 일반 가이드 수준으로 표현
  3. 경고는 과다 가능성 또는 주의 필요 수준으로만 표현

[공통 강제 규칙]
- 로그인, 관리자, 결제, 문의, 운영 기능 화면은 넣지 말아줘.
- 화면 수는 최소 3개, 최대 5개 안에서 고정해줘.
- 사용자가 가장 짧게 입력 -> 분석 -> 결과를 체험할 수 있는 구조를 우선해줘.
- 결과 화면은 반드시 입력 목록 / 중복 성분 / 과다 가능성 / 복용 시간 가이드 / 요약 한 줄을 포함해줘.
- 추천은 자동 승계하지 말고 확정만 다음 단계로 넘겨줘.
- Prototype Decision Register와 P1 authoritative_updates를 최우선으로 사용해줘.
- 이후 단계에서 route 이름, 화면 이름, 핵심 CTA 이름이 바뀌지 않도록 처음부터 단순하게 고정해줘.
- 마지막에 반드시 아래 7개를 포함해줘.
  1. 이번 단계 확정 사항
  2. 다음 단계 입력으로 넘길 핵심 요약
  3. 추가 확인 필요 항목
  4. 충돌 가능 항목
  5. 바로 다음 단계 진행 가능 항목
  6. 단계 상태
  7. authoritative_updates YAML
